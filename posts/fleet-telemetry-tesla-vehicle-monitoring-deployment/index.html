使用 Fleet Telemetry 让 Tesla 车辆主动上报数据进行监控-(三)部署服务

深入解析使用 Fleet Telemetry 监控特斯拉车辆数据，涵盖 tesla-http-proxy、EMQX MQTT 部署及核心配置详解。

2025-09-26 | Updated 2026-07-11

  • Tesla

一、部署顺序说明

Fleet Telemetry 的数据链路是车辆直接连接 Fleet Telemetry 服务，tesla-http-proxy 只在下发配置时使用，不在车辆上报链路中。因此部署顺序建议为：

  1. 准备 Fleet Telemetry records 使用的 datastore，例如 MQTT、Kafka、Pub/Sub、Kinesis 或 logger。
  2. 部署 Fleet Telemetry，使车辆可以通过公网访问 hostname:port。
  3. 使用 check_server_cert.sh 校验将要下发给车辆的 hostname、port、ca 是否和服务端 TLS 证书匹配。
  4. 确认虚拟钥匙已经注册并添加到车辆。
  5. 部署或启动 tesla-http-proxy，通过它调用 Fleet API 下发配置。

二、准备 MQTT datastore

如果 Fleet Telemetry 配置使用 MQTT dispatcher，需要先准备一个 MQTT broker。MQTT 可以使用 EMQX、Mosquitto 等服务；因为 EMQX 提供消息处理能力，所以这里以 EMQX 为例进行部署，详细可以参考 通过 Docker 运行 EMQX。

  • docker-compose.yaml
services:
  mqtt:
    image: emqx/emqx-enterprise
    container_name: mqtt
    restart: always
    ports:
      - "1883:1883" # MQTT
      - "8083:8083" # WebSocket
      - "8084:8084" # WebSocket/TLS
      - "8883:8883" # MQTT/TLS
      - "18083:18083" # Dashboard
    volumes:
      - ./data:/opt/emqx/data
      - ./log:/opt/emqx/log

如果只需要一个最小 MQTT broker，也可以使用 Mosquitto：

  • mosquitto.conf
listener 1883
allow_anonymous true
  • docker-compose.yaml
services:
  mqtt:
    image: eclipse-mosquitto:2
    container_name: mqtt
    restart: always
    ports:
      - "1883:1883" # MQTT
    volumes:
      - ./mosquitto.conf:/mosquitto/config/mosquitto.conf:ro

三、部署 Fleet Telemetry

3.1 配置文件

Fleet Telemetry 需要一个配置文件，用于指定监听的端口、转发的消息等组件

  • config.json

关于配置的详细的配置解释可以参考 Install steps

这里将消息转发给 MQTT，监听 12345 端口，TLS 使用第一篇中通过 Certbot 从公开 CA 申请的服务端证书。MQTT 的车辆字段 topic 格式为 <topic_base>/<VIN>/v/<field_name>，如车速为 telemetry/xxx/v/VehicleSpeed；连接状态 topic 是 <topic_base>/<VIN>/connectivity，详细可以参考 MQTT Datastore

这几个证书不要混：

  • tls.server_cert: Fleet Telemetry 服务端证书，本文填 /certs/fullchain.pem。
  • tls.server_key: 服务端证书对应的私钥，本文填 /certs/privkey.pem。
  • fleet_telemetry_config.ca: 下发给车辆的 CA 链，Certbot 场景通常取 certs/tls/chain.pem 的内容。

车辆客户端证书由 Tesla 车辆侧提供和管理，部署者不自行生成。tls.ca_file 是高级场景下追加客户端证书 CA 的配置，不是下发给车辆的 ca 字段。

3.2 部署 Fleet Telemetry

使用 docker 部署 Fleet Telemetry，目录结构如下：

.
├── certs
│   └── tls
│       ├── cert.pem
│       ├── chain.pem
│       ├── fullchain.pem
│       └── privkey.pem
├── config.json
├── docker-compose.yaml
  • docker-compose.yaml

需要挂载前面使用 Certbot 申请的服务端 TLS 证书和私钥。Fleet Telemetry 和车辆使用 mTLS 进行通信，如果中间有反向代理或负载均衡，需要保证客户端证书校验仍然在 Fleet Telemetry 服务或等价入口上正确完成；普通 HTTPS 反向代理会破坏这条链路。排查阶段建议先让 Fleet Telemetry 端口直接公网可达，验证通过后再考虑更复杂的入口层。

services:
  fleet-telemetry:
    image: tesla/fleet-telemetry:latest
    container_name: fleet-telemetry
    command:
      - /fleet-telemetry
      - -config=/etc/fleet-telemetry/config.json
    ports:
      - 12345:12345 # TLS port
      - 12346:12346 # status port
      - 9091:9091 # prometheus metrics port
    volumes:
      - ./config.json:/etc/fleet-telemetry/config.json:ro
      - ./certs/tls/fullchain.pem:/certs/fullchain.pem:ro
      - ./certs/tls/privkey.pem:/certs/privkey.pem:ro

启动后会输出相关的监听日志:

fleet-telemetry  | 2025/09/27 06:15:02 maxprocs: Leaving GOMAXPROCS=1: CPU quota undefined
fleet-telemetry  | {"activity":true,"context":"fleet-telemetry","level":"info","msg":"starting_server","time":"2025-09-27T06:15:02Z"}
fleet-telemetry  | {"activity":true,"context":"fleet-telemetry","level":"info","msg":"profiler_started","port":4269,"time":"2025-09-27T06:15:02Z"}

访问 status 端口检查状态，返回 ok 说明服务正确启动

curl localhost:12346/status
ok#

3.3 验证服务端证书链

检查时需要通过 validate_server.json 声明将要下发给车辆的 hostname、port 和 ca。这个文件不是 Fleet Telemetry 服务端配置，只是给官方 check_server_cert.sh 使用。

ca 填签发 Fleet Telemetry 服务端证书的 CA 链。按本文的 Certbot / Let’s Encrypt 示例，先用 certs/tls/chain.pem，把它转成带 \n 的单行字符串后写进 validate_server.json。不要填 fullchain.pem、cert.pem、privkey.pem，也不要填虚拟钥匙。

awk '{printf "%s\\n",$0}' certs/tls/chain.pem
  • validate_server.json
{
  "hostname": "fleet.example.com",
  "port": 12345,
  "ca": "-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----\n"
}

执行 check_server_cert.sh 脚本检查配置。如果配置正确将返回 OK 状态，说明下发配置里的 hostname、port、ca 与服务端 TLS 证书匹配。如果脚本只提示 partial chain warning，应补齐 CA 链后再次验证。

./check_server_cert.sh validate_server.json
/tmp/tmp.yhc6u2DX4Y: OK
The server certificate is valid.

如果不确定当前服务端实际返回的是哪条证书链，可以直接从已经部署的 Fleet Telemetry 入口提取。下面的命令会连接 hostname:port，取服务端返回的证书链，去掉第一张服务端证书，只保留后面的 CA chain，并生成 validate_server.json：

HOST="fleet.example.com"
PORT="12345"

openssl s_client -showcerts -connect "${HOST}:${PORT}" -servername "${HOST}" </dev/null 2>/dev/null \
| awk '/BEGIN CERTIFICATE/{i++} i>1 {print} /END CERTIFICATE/{if (i>1) print ""}' \
> server-ca-chain.pem

jq -Rs --arg host "${HOST}" --argjson port "${PORT}" \
  '{hostname: $host, port: $port, ca: .}' \
  server-ca-chain.pem \
> validate_server.json

./check_server_cert.sh validate_server.json

如果 server-ca-chain.pem 是空文件，通常说明服务端没有发送中间证书链，应优先检查 Fleet Telemetry 的 server_cert 是否使用了完整证书链。按 Certbot / Let’s Encrypt 的文件命名，服务端 server_cert 应使用 fullchain.pem，不是只有站点证书的 cert.pem。

这个检查不能证明虚拟钥匙、车辆固件、应用注册、字段配置都正确；它只验证车辆后续能否用配置中的 ca 校验 Fleet Telemetry 服务端证书。

四、部署 tesla-http-proxy

tesla-http-proxy 可以使用 https://github.com/helloworlde/fleet-telemetry-proxy 在 Cloudflare Worker 部署替代

vehicle-command 中的 tesla-http-proxy 用于签名并转发 Fleet API 请求。下发 Fleet Telemetry 配置时，proxy 会使用虚拟钥匙私钥对配置签名，再转发到 Tesla Fleet API。车辆端会用已经通过 _ak 加入 keychain 的应用公钥验签；如果只注册了 Partner Account，但没有把虚拟钥匙添加到车辆，下发配置可能因 missing_key 被跳过。

tesla-http-proxy 不参与车辆到 Fleet Telemetry 的上报链路，不建议直接暴露到公网。官方工具在非 localhost 监听时也会提示需要额外客户端认证，否则未授权客户端可能造成异常流量。个人部署建议只绑定本机或内网地址。

文件目录结构如下，fleet-private-key.pem 是第二篇中自己用 OpenSSL 生成的虚拟钥匙私钥，只给 tesla-http-proxy 签名请求使用；tls/fullchain.pem 是第一篇中由公开 CA 签发的服务端 TLS 证书链，tls/privkey.pem 是与它配套的本地私钥：

.
├── certs
│   ├── fleet-private-key.pem
│   └── tls
│       ├── fullchain.pem
│       └── privkey.pem
└── docker-compose.yaml
  • docker-compose.yaml
services:
  tesla_http_proxy:
    image: tesla/vehicle-command:latest
    container_name: tesla_http_proxy
    ports:
      - 127.0.0.1:4443:4443
    environment:
      - TESLA_HTTP_PROXY_TLS_CERT=/certs/tls/fullchain.pem
      - TESLA_HTTP_PROXY_TLS_KEY=/certs/tls/privkey.pem
      - TESLA_KEY_FILE=/certs/fleet-private-key.pem
      - TESLA_HTTP_PROXY_HOST=0.0.0.0
      - TESLA_HTTP_PROXY_PORT=4443
      - TESLA_HTTP_PROXY_TIMEOUT=10s
      - TESLA_VERBOSE=true
    security_opt:
      - no-new-privileges:true
    volumes:
      - ./certs/tls/fullchain.pem:/certs/tls/fullchain.pem:ro
      - ./certs/tls/privkey.pem:/certs/tls/privkey.pem:ro
      - ./certs/fleet-private-key.pem:/certs/fleet-private-key.pem:ro

启动后可以在本机检查健康状态：

curl --resolve fleet.example.com:4443:127.0.0.1 https://fleet.example.com:4443/health
OK

如果 proxy 使用的是本地自签名证书，这张自签名证书也是部署者自己生成、只用于访问 proxy 的本地 TLS 身份；需要再给 curl 指定该 proxy 证书对应的 CA 或证书文件。不要把这里的 --cacert 和下发给车辆的 Fleet Telemetry ca 混为一谈。
