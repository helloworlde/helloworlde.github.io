使用 Fleet Telemetry 让 Tesla 车辆主动上报数据进行监控-(四)下发配置

Tesla Fleet Telemetry 监控：获取 access_token 鉴权及通过 tesla-http-proxy 下发车辆配置与数据上报策略详解

2025-09-27 | Updated 2026-07-11

  • Tesla

一、配置车辆

配置车辆时需要使用鉴权，因此需要先获取 access_token

1.1 获取 access_token

access_token 可以参考下面的方式手动获取，也可以使用 https://github.com/helloworlde/tesla-access-token 项目在 Cloudflare 部署 Worker 进行获取(注意：这个 Worker 绝对不能公开访问，使用前务必通过 Cloudflare Access 配置鉴权策略，保证仅自己可见)

需要用户授权的 access_token ，用于调用 tesla-http-proxy 向车辆下发配置；这里的 access_token 类型是 ‘第三方令牌’，详细参考 第三方令牌

  • 用户授权

首先需要使用之前申请的应用的 client_id 和 redirect_uri 访问下面的链接，进行用户授权；需要替换 ${CLIENT_ID}、${REDIRECT_URI} 和 ${STATE}，其中 ${STATE} 可以使用随机字符串，如 123456 等。这里的 scope 按最小权限填写：下发 Fleet Telemetry 配置和接收车辆数据需要 vehicle_cmds、vehicle_device_data；如果配置了 Location、导航路线等位置相关字段，再额外添加 vehicle_location。

https://auth.tesla.cn/oauth2/v3/authorize?client_id=${CLIENT_ID}&locale=zh-CN&prompt=login&redirect_uri=${REDIRECT_URI}&response_type=code&scope=openid%20offline_access%20user_data%20vehicle_device_data%20vehicle_cmds&state=${STATE}

然后会进入登录页面，成功登录后会跳转到 ${REDIRECT_URI}，并且携带 code 参数，这个 code 用于获取 access_token:

${REDIRECT_URI}?locale=zh-CN&code=CN_xxxx&state=${STATE}&issuer=https%3A%2F%2Fauth.tesla.cn%2Foauth2%2Fv3

  • 获取 access_token

使用上面的请求参数中的 code，访问 代码交换 接口获取 access_token

需要替换 ${CLIENT_ID}、${CLIENT_SECRET}、${REDIRECT_URI} 和 ${CODE}，其中 ${CODE} 是上面获取到的 code

curl --location 'https://auth.tesla.cn/oauth2/v3/token' \
--header 'Content-Type: application/x-www-form-urlencoded' \
--data-urlencode 'grant_type=authorization_code' \
--data-urlencode 'client_id=${CLIENT_ID}' \
--data-urlencode 'client_secret=${CLIENT_SECRET}' \
--data-urlencode 'audience=https://fleet-api.prd.cn.vn.cloud.tesla.cn' \
--data-urlencode 'redirect_uri=${REDIRECT_URI}' \
--data-urlencode 'scope=openid offline_access user_data vehicle_device_data vehicle_cmds' \
--data-urlencode 'code=${CODE}'

响应中的 access_token 就是用于鉴权的令牌，有效期通常是 8 个小时。因为授权请求包含 offline_access，响应中还会返回 refresh_token。刷新令牌需要安全保存，并且是单次使用；每次用 refresh token 换取新 token 后，都要保存新的 refresh token。官方文档说明 refresh token 有效期为 3 个月，最近一次使用过的 refresh token 在异常情况下最多还能短暂兼容 24 小时。

{
  "access_token": "eyXXXX",
  "id_token": "eyXXXX",
  "refresh_token": "CN_xxxx",
  "expires_in": 28800,
  "state": "xxxxx",
  "token_type": "Bearer"
}

1.2 下发配置

下发配置需要调用 Fleet 车辆端点的 API。推荐通过 tesla-http-proxy 调用 fleet_telemetry_config create，proxy 会使用应用的虚拟钥匙私钥对配置签名并转发到 Fleet API。车辆会用已经通过 _ak 加入 keychain 的应用公钥验签；如果虚拟钥匙没有添加到车辆，响应里可能出现 missing_key。也可以自己构造签名后的 JWS 调用 fleet_telemetry_config_jws，但官方不推荐直接使用该接口。

配置的内容是一个 JSON 文件，指定需要上报的数据和上报的频率等信息，相关的字段可以参考 protos/vehicle_data.proto

  • 准备 ca

ca 是车辆用来验证 Fleet Telemetry 服务端证书的 CA 链。按本文的 Certbot / Let’s Encrypt 示例，先用 certs/tls/chain.pem，把它转换成带 \n 的单行字符串后填到请求体。不要填 fullchain.pem、cert.pem、privkey.pem，也不要填虚拟钥匙。

awk '{printf "%s\\n",$0}' certs/tls/chain.pem

输出内容格式如下：

-----BEGIN CERTIFICATE-----\nxxxxxxxxx\n-----END CERTIFICATE-----

下发前建议先用第三篇的 validate_server.json 和 check_server_cert.sh 校验：

{
  "hostname": "fleet.example.com",
  "port": 12345,
  "ca": "-----BEGIN CERTIFICATE-----\nxxxxxxxxx\n-----END CERTIFICATE-----\n"
}

只有这个校验明确通过，才说明车辆收到同样的 hostname、port、ca 后可以验证 Fleet Telemetry 服务端证书。如果脚本只提示 partial chain warning，应补齐 CA 链后再次验证。

  • 下发配置

下发的配置中需要指定车辆的 VIN 码、监听的端口、上报的数据和频率等信息，该接口的解释说明参考 fleet_telemetry_config create

其中的 fields 字段指定需要上报的数据和上报的频率等信息，详细可以参考 protos/vehicle_data.proto，完整的字段配置和说明可以参考 使用 Fleet Telemetry 让 Tesla 车辆主动上报数据进行监控-(七)完整配置说明 以及 可用数据

这里仅配置速度、里程、电量和档位等字段，其他字段可以根据需要添加。interval_seconds 指定同一字段最小上报间隔；信号基于变化触发，但不会比该间隔更频繁。minimum_delta 指定数值或位置字段变化的最小阈值，适合减少高频小幅抖动。Fleet Telemetry client 1.1.0 起已经移除 prefer_typed，所有数据现在都以 typed 形式返回，不需要再配置 prefer_typed。

curl --location 'https://fleet.example.com:4443/api/1/vehicles/fleet_telemetry_config' \
--resolve fleet.example.com:4443:127.0.0.1 \
--header 'Authorization: Bearer ${ACCESS_TOKEN}' \
--header 'Content-Type: application/json' \
--data '{
    "config": {
        "delivery_policy": "latest",
        "port": 12345,
        "alert_types": [
            "service",
            "customer",
            "service-fix"
        ],
        "fields": {
            "VehicleSpeed": {
                "interval_seconds": 5,
                "minimum_delta": 0.01
            },
            "Odometer": {
                "interval_seconds": 600,
                "minimum_delta": 0.1
            },
            "Soc": {
                "interval_seconds": 300,
                "minimum_delta": 0.1
            },
            "Gear": {
                "interval_seconds": 5
            }
        },
        "ca": "-----BEGIN CERTIFICATE-----\nxxxxxxxxx\n-----END CERTIFICATE-----\n",
        "hostname": "fleet.example.com"
    },
    "vins": [
        "{{VIN}}"
    ]
}'

下发配置成功后会返回响应, 更新了一辆车的配置：

{
  "response": {
    "updated_vehicles": 1
  }
}

1.3 获取配置

可以通过下面的接口获取当前车辆的配置，参考 fleet_telemetry_config get

curl --location 'https://fleet.example.com:4443/api/1/vehicles/${VIN}/fleet_telemetry_config' \
--resolve fleet.example.com:4443:127.0.0.1 \
--header 'Authorization: Bearer ${ACCESS_TOKEN}'

当返回的响应中 synced 字段为 true，说明车辆已经采用目标配置，config 字段中是当前车辆的配置内容。synced=false 表示车辆会在下次建立后端连接时尝试采用目标配置，不一定代表配置错误；如果长时间不同步，需要检查车辆网络、虚拟钥匙是否已添加到车辆、固件版本、Fleet Telemetry 服务端证书链和 fleet_telemetry_errors。

{
  "response": {
    "synced": true,
    "config": {
      "hostname": "fleet.example.com",
      "ca": "-----BEGIN CERTIFICATE-----\nxxxxxxxxx\n-----END CERTIFICATE-----\n",
      "port": 12345,
      "delivery_policy": "latest",
      "fields": {
        "VehicleSpeed": {
          "interval_seconds": 5,
          "minimum_delta": 0.01
        },
        "Odometer": {
          "interval_seconds": 600,
          "minimum_delta": 0.1
        },
        "Soc": {
          "interval_seconds": 300,
          "minimum_delta": 0.1
        },
        "Gear": {
          "interval_seconds": 5
        }
      },
      "alert_types": ["service", "customer", "service-fix"],
      "iss": "xxx",
      "aud": "com.tesla.fleet.TelemetryClient"
    },
    "limit_reached": false,
    "key_paired": true
  }
}

如果车辆没有开始上报，可以查询最近的 Fleet Telemetry 错误：

curl --location 'https://fleet.example.com:4443/api/1/vehicles/${VIN}/fleet_telemetry_errors' \
--resolve fleet.example.com:4443:127.0.0.1 \
--header 'Authorization: Bearer ${ACCESS_TOKEN}'

上面的请求访问的是本机绑定的 tesla-http-proxy，--resolve 只是让 fleet.example.com:4443 指向 127.0.0.1，同时保留 TLS 校验时使用的域名。如果 proxy 使用本地自签名证书，再额外指定该 proxy 证书对应的 --cacert；这里不要复用下发给车辆的 Fleet Telemetry ca。

1.4 没有收到上报时的排查顺序

下发接口返回成功只表示 Tesla 后端接受了目标配置，不代表车辆已经连上 Fleet Telemetry 服务。建议按下面顺序排查。

第一步，重新获取车辆配置，确认关键字段：

  • synced=true: 车辆已经采用目标配置；如果还是 false，先等待车辆建立后端连接，或检查车辆网络、固件版本和虚拟钥匙。
  • key_paired=true: 车辆已经配对当前应用虚拟钥匙；如果不是 true，优先回到第二篇重新添加虚拟钥匙。
  • limit_reached=false: 车辆没有达到 Fleet Telemetry streaming 应用数量上限。
  • config.hostname、config.port、config.ca: 必须和实际公网 Fleet Telemetry 入口一致。

第二步，查询 Partner 级错误。中国区 Fleet API 直接使用 https://fleet-api.prd.cn.vn.cloud.tesla.cn，这里需要 Partner token：

PARTNER_TOKEN="你的 Partner token"

curl -sS \
  --request GET \
  --header "Authorization: Bearer ${PARTNER_TOKEN}" \
  --header "Content-Type: application/json" \
  "https://fleet-api.prd.cn.vn.cloud.tesla.cn/api/1/partner_accounts/fleet_telemetry_error_vins" \
| jq

curl -sS \
  --request GET \
  --header "Authorization: Bearer ${PARTNER_TOKEN}" \
  --header "Content-Type: application/json" \
  "https://fleet-api.prd.cn.vn.cloud.tesla.cn/api/1/partner_accounts/fleet_telemetry_errors" \
| jq

fleet_telemetry_error_vins 返回 null 只表示当前没有可列出的错误 VIN，不代表车辆已经成功连接。真正定位问题应看 fleet_telemetry_errors 中的具体错误内容。

第三步，根据错误内容处理。

如果看到下面的错误：

webconnection error: tls: failed to verify certificate: x509: certificate signed by unknown authority

说明车辆无法用配置里的 ca 验证 hostname:port 当前返回的服务端 TLS 证书。直接从当前服务端提取 CA chain，再生成校验文件：

HOST="fleet.example.com"
PORT="12345"

openssl s_client -showcerts -connect "${HOST}:${PORT}" -servername "${HOST}" </dev/null 2>/dev/null \
| awk '/BEGIN CERTIFICATE/{i++} i>1 {print} /END CERTIFICATE/{if (i>1) print ""}' \
> server-ca-chain.pem

jq -Rs --arg host "${HOST}" --argjson port "${PORT}" \
  '{hostname: $host, port: $port, ca: .}' \
  server-ca-chain.pem \
> validate_server.json

./check_server_cert.sh validate_server.json

只有输出 The server certificate is valid. 后，才把同一个 server-ca-chain.pem 写回下发配置的 ca 字段。不要手工复制 jq -Rs . server-ca-chain.pem 的输出再塞进另一个 JSON 字符串，容易造成双重转义。推荐用 jq --rawfile 注入：

jq --rawfile ca server-ca-chain.pem '.config.ca = $ca' \
  fleet-telemetry-config.json \
> fleet-telemetry-config.fixed.json

如果下发时返回：

ca is not a valid PEM

说明 API 收到的 ca 不是可解析的 PEM。常见原因是多包了一层引号，或者把 \n 写成了普通文本。用下面的命令检查最终请求体中的 ca：

jq -r '.config.ca' fleet-telemetry-config.fixed.json \
| openssl x509 -noout -subject -issuer

如果看到下面的错误：

error loading config file: InvalidSignature: unknown issuer

说明车辆不认识签名这份配置的虚拟钥匙。重点检查三件事：

  • tesla-http-proxy 使用的 fleet-private-key.pem 是否就是当前应用公钥对应的私钥。
  • 托管在 /.well-known/appspecific/com.tesla.3p.public-key.pem 的公钥是否由这把私钥导出。
  • 用户是否已经通过 _ak 链接把这个公钥添加到这辆车的 keychain。

如果 synced=true、Partner 错误里没有新的证书或签名错误，但 Fleet Telemetry 服务仍没有日志，再检查公网入口和服务端部署：

  • hostname:port 是否能从公网访问，不只是本机或内网可访问。
  • 是否有 Cloudflare、Nginx、负载均衡器提前终止 TLS，导致 Fleet Telemetry 收不到车辆客户端证书。
  • Fleet Telemetry 的日志里是否出现 TLS handshake 或 connectivity 记录。
  • 配置的 records 是否至少把 connectivity、V、alerts、errors 输出到 logger，方便先确认连接是否建立。
