使用 Fleet Telemetry 让 Tesla 车辆主动上报数据进行监控-(二)添加虚拟钥匙

Tesla Fleet Telemetry 监控教程：开发者账号创建、公钥私钥生成及虚拟钥匙车辆绑定步骤详解

2025-09-25 | Updated 2026-03-22

  • Tesla

让 Tesla 主动上报数据到 Fleet Telemetry，需要先让车辆信任应用的虚拟钥匙。虚拟钥匙是一组 prime256v1 公私钥：公钥托管在应用域名下并注册到 Tesla，私钥保存在自己的服务端，供 tesla-http-proxy 对 Fleet Telemetry 配置进行签名。详细操作可以参考 什么是 Fleet API

一、申请开发者账号和应用

创建开发者账号需要 Tesla 账号设置中绑定邮箱，并且开启两步验证；然后访问 Tesla Developer 创建应用，输入应用名称和说明等信息

OAuth 授权类型选择 “授权码和机器对机器”，然后添加允许的源 URL、重定向 URL 和返回的 URL。这里需要区分几个 URL：

  • 应用域名：用于托管 /.well-known/appspecific/com.tesla.3p.public-key.pem，并在 Partner Account register 时注册
  • OAuth redirect URI：用于用户授权回调，必须和开发者控制台、授权请求中的 redirect_uri 一致
  • Fleet Telemetry hostname：车辆后续上报数据的入口，写入 fleet_telemetry_config 的 hostname 字段

这几个地址可以使用同一个域名，但职责不同，不要求 OAuth redirect URI 等于 Fleet Telemetry 的上报地址。

选择权限范围时按实际需要勾选，不建议直接选择所有权限。Fleet Telemetry 下发配置和接收车辆数据通常需要 vehicle_cmds、vehicle_device_data；如果要上报 Location、导航路线等位置相关字段，还需要 vehicle_location。

创建成功后会得到 Client ID 和 Client Secret，需要保存下来，后续获取 Token 需要使用

二、生成公钥/私钥对

需要生成一个公钥/私钥对，公钥需要托管在网站上让 Tesla 访问，用于注册应用和添加虚拟钥匙；私钥用于 tesla-http-proxy 对配置请求签名。这个公私钥对不是服务端 TLS 证书，不能和 fullchain.pem / privkey.pem 混用。

这组虚拟钥匙由自己在本地生成。fleet-private-key.pem 不是 CA、Cloudflare 或 Tesla 签发的证书；com.tesla.3p.public-key.pem 只是从私钥导出的公钥。后续 Partner Account register 是把域名和公钥登记到 Tesla，不是让 Tesla 签发这组密钥。

对应关系只有这一组：

fleet-private-key.pem              # 虚拟钥匙私钥，自己保存，给 tesla-http-proxy 签名配置/命令
com.tesla.3p.public-key.pem        # 虚拟钥匙公钥，公开托管，给 Tesla 下载，并通过 _ak 加入车辆 keychain

它们的完整作用链路是：先把公钥托管并通过 Partner Account register 登记域名；再让用户打开 _ak 链接，把这个公钥作为虚拟钥匙添加到车辆；之后通过 tesla-http-proxy 下发 Fleet Telemetry 配置时，proxy 使用 fleet-private-key.pem 签名配置，车辆用 keychain 里的公钥验证签名。如果只完成 register 但没有通过 _ak 添加到车辆，下发配置时车辆仍可能返回 missing_key。

Tesla 下载的 URL 必须只返回 com.tesla.3p.public-key.pem 这个小文件，内容应该是 BEGIN PUBLIC KEY。不要把 TLS 证书、CA 链、fullchain.pem、cert.pem、私钥或 HTML 页面放到这个地址；否则 Partner Account register 可能返回 Public key download failed ... error: too large。

  • 生成私钥
openssl ecparam -name prime256v1 -genkey -noout -out fleet-private-key.pem
  • 生成公钥
openssl ec -in fleet-private-key.pem -pubout -out com.tesla.3p.public-key.pem

可以先在本地检查公钥文件：

cat com.tesla.3p.public-key.pem
wc -c com.tesla.3p.public-key.pem

正常内容类似下面这样，大小通常不到 200 字节：

-----BEGIN PUBLIC KEY-----
...
-----END PUBLIC KEY-----

三、托管公钥

然后需要将公钥托管在应用域名的 /.well-known/appspecific/ 路径下，确保可以通过 https://fleet.example.com/.well-known/appspecific/com.tesla.3p.public-key.pem 访问到公钥文件。Tesla 后端还可能使用 Range: bytes=0-200 请求校验公钥，因此不要做需要登录、跳转或动态鉴权的托管。

可以使用 Caddy 进行托管。这里 Caddy 只用于提供 HTTPS 静态文件，不参与 Fleet Telemetry 的车辆上报链路。需要将 com.tesla.3p.public-key.pem 文件复制到 static/com.tesla.3p.public-key.pem，然后挂载到 Caddy 的 /var/www/html/.well-known/appspecific/ 目录下。

  • docker-compose.yaml
services:
  caddy:
    image: caddy
    container_name: caddy
    hostname: caddy
    ports:
      - "443:443"
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - ./certs/tls:/certs/tls:ro # Certbot 生成的服务端 TLS 证书
      - ./static:/var/www/html/.well-known/appspecific/:ro # 用于托管公钥
    restart: unless-stopped
  • Caddyfile

创建 Caddyfile 文件，内容如下：

fleet.example.com {
    # 这里使用的是第一篇中 Certbot 从公开 CA 申请的服务端 TLS 证书，不是虚拟钥匙
    tls /certs/tls/fullchain.pem /certs/tls/privkey.pem

    # 挂载的目录
    root * /var/www/html

    route {
        # 处理特定文件
        handle /.well-known/appspecific/* {
            file_server
        }
    }
}

完整的目录结构如下：

.
├── Caddyfile
├── certs
│   └── tls
│       ├── fullchain.pem
│       └── privkey.pem
├── fleet-private-key.pem
├── docker-compose.yaml
└── static
    └── com.tesla.3p.public-key.pem

启动容器，然后验证公钥可以公开访问：

curl https://fleet.example.com/.well-known/appspecific/com.tesla.3p.public-key.pem
curl -H 'Range: bytes=0-200' https://fleet.example.com/.well-known/appspecific/com.tesla.3p.public-key.pem
curl -s https://fleet.example.com/.well-known/appspecific/com.tesla.3p.public-key.pem | wc -c

前两个请求都能返回 PEM 格式公钥，并且大小是一个很小的 P-256 public key，才说明托管方式满足后续注册和配对要求。如果返回 404/HTML/跳转页面、证书链或私钥，Tesla 都可能认为公钥下载失败。

四、注册应用域名和公钥

公钥可以公开访问之后，还需要调用 Partner Account register 接口，将应用域名和公钥注册到 Tesla。这个步骤只是让 Tesla 知道“这个域名对应哪个公钥”，还不会把公钥加入车辆；真正让车辆信任该公钥的是下一步 _ak 添加虚拟钥匙。否则打开 _ak 链接时可能提示应用未注册，或者车辆无法识别该虚拟钥匙。

注册步骤需要使用 Partner Authentication Token，和用户授权得到的第三方 access token 不是同一种令牌。注册时的 domain 必须与开发者控制台 allowed origins 的根域匹配，并且该域名下必须能访问公钥文件。例如应用域名为 fleet.example.com 时，需要保持下面的 URL 可访问：

https://fleet.example.com/.well-known/appspecific/com.tesla.3p.public-key.pem

调用中国区 Fleet API 时，可以按下面的方式注册：

curl --location 'https://fleet-api.prd.cn.vn.cloud.tesla.cn/api/1/partner_accounts' \
--header 'Authorization: Bearer ${PARTNER_AUTHENTICATION_TOKEN}' \
--header 'Content-Type: application/json' \
--data '{
  "domain": "fleet.example.com"
}'

注册完成后，可以查询域名对应的公钥确认注册成功：

curl --location 'https://fleet-api.prd.cn.vn.cloud.tesla.cn/api/1/partner_accounts/public_key?domain=fleet.example.com' \
--header 'Authorization: Bearer ${PARTNER_AUTHENTICATION_TOKEN}'

确认 public key 返回正常后，再继续通过 Tesla App 将虚拟钥匙添加到车辆。

五、虚拟密钥添加到车辆

当公钥托管并注册成功之后，就可以访问 https://www.tesla.com/_ak/fleet.example.com，将虚拟密钥添加到车辆。用户需要使用已经授权该应用的同一个 Tesla 账号登录 Tesla App，然后选择对应车辆，添加虚拟钥匙；添加成功后，车辆的钥匙中会出现 fleet.example.com 虚拟钥匙。

这一步的结果是车辆把应用公钥加入自己的 keychain。后续 tesla-http-proxy 用 fleet-private-key.pem 签名 Fleet Telemetry 配置或车辆命令时，车辆会用这个已添加的公钥验签。

如果账号下有多辆车，也可以带上 VIN 参数：

https://www.tesla.com/_ak/fleet.example.com?vin=${VIN}

如果添加失败，优先检查：

  • 用户是否已授权应用，并授予 vehicle_device_data、vehicle_cmds；如果配置位置字段，还需要 vehicle_location
  • Tesla App 登录账号是否和授权账号一致
  • 应用域名和公钥是否已经通过 Partner Account register 注册
  • 公钥是否仍然可以通过 /.well-known/appspecific/com.tesla.3p.public-key.pem 访问
  • 车辆是否已经通过 _ak 成功添加该虚拟钥匙；否则下发 Fleet Telemetry 配置可能返回 missing_key
  • 车辆是否满足 Fleet Telemetry 的固件和硬件要求
