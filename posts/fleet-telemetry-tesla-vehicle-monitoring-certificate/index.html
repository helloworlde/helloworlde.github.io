使用 Fleet Telemetry 让 Tesla 车辆主动上报数据进行监控-(一)申请证书

特斯拉 Fleet Telemetry 主动上报数据监控方案，基于 mTLS 证书申请与部署指南，替代 Owner API 实现车队管理。

2025-09-24 | Updated 2026-07-11

  • Tesla

Tesla Fleet Telemetry 是 Tesla 官方的 telemetry protocol 参考实现，可以让车辆通过 mTLS WebSocket 主动上报数据到自托管 Fleet Telemetry 服务。通过下发 fleet_telemetry_config，应用可以控制车辆连接的 hostname、port、证书链和需要上报的字段。

服务 Tessie 就是基于 Fleet Telemetry 的数据构建的看板和分析统计工具：

Fleet Telemetry 服务负责接收车辆连接、校验车辆客户端证书、处理 ack/error/rate limit，并把车辆数据、连接状态、警报和错误分发到配置的 datastore，例如 MQTT、Kafka、Pub/Sub、Kinesis 或 logger。Fleet Telemetry 本身不是数据库，也不负责业务侧数据建模。

Fleet Telemetry 能获取到的数据和 Owner API 基本是一样的，仅在格式和上报方式上有区别；因此只要适当处理数据格式，也可以兼容 TeslaMate 等工具。Fleet Telemetry 可上报字段、字段类型和可用性会随车型、硬件、固件和 Fleet Telemetry client 版本变化，车辆也可能对不适用或暂不可测量的字段返回 invalid: true。详细的数据和警报可以参考 可用数据 和 车辆警报

一、前置条件

开始部署前，先确认车辆、账号、域名和证书满足 Fleet Telemetry 的基本要求：

项目要求
车辆和固件Fleet Telemetry 基础支持需要车辆固件 2023.20.6 或更高版本；通过 tesla-http-proxy 下发配置时，官方接口会拒绝低于 2024.26 的车辆；Intel Atom 旧款 Model S/X 还需要 2025.20 或更高版本。部分旧款 Model S/X 不支持。
应用注册和虚拟钥匙应用公钥必须已经通过 Partner Account register 注册，并作为虚拟钥匙添加到车辆。
用户授权下发 Fleet Telemetry 配置和接收车辆数据通常需要 vehicle_cmds、vehicle_device_data；如果配置位置相关字段，还需要用户授权 vehicle_location。不要请求无关权限。
服务入口Fleet Telemetry 服务端必须能被车辆从公网访问，且 hostname、port、ca 要能通过官方 check_server_cert.sh 校验。
数量限制fleet_telemetry_config create 可能因车辆已有配置过多返回 max_configs；fleet_telemetry_config get 中 limit_reached=true 表示车辆已达到 Fleet Telemetry streaming 应用数量上限，不能再添加新的 streaming 配置。

二、整体关系

Fleet Telemetry 部署中容易混淆的是域名、证书、密钥和服务职责。建议先按下面的关系理解，再进入后续操作：

名称示例作用调用方是否需要公网
应用域名 / 公钥托管域名fleet.example.com托管 /.well-known/appspecific/com.tesla.3p.public-key.pem，供 Tesla 校验应用公钥Tesla 后端、Tesla App是，通常使用 443
Fleet Telemetry 上报入口fleet.example.com:12345车辆通过 mTLS WebSocket 连接并主动上报数据车辆是
tesla-http-proxy 管理入口127.0.0.1:4443 或内网地址对 Fleet API 请求签名并转发，用于下发配置自己的脚本或管理服务不建议公网暴露
Datastore / dispatcherMQTT/Kafka/PubSub/Kinesis/logger接收 Fleet Telemetry 分发的 recordsFleet Telemetry 服务按实际部署决定

同一个域名可以同时用于公钥托管和 Fleet Telemetry 上报入口，但它们是不同用途：443 通常用于托管公钥；Fleet Telemetry 可以监听 443，也可以监听其他公网端口。tesla-http-proxy 不是车辆上报入口，只用于向 Fleet API 下发签名后的配置请求。

证书和密钥

先记住一句话：ca 填“签发 Fleet Telemetry 服务端证书的 CA 链”。如果按本文用 Certbot / Let’s Encrypt 申请证书，先用 certs/tls/chain.pem 的内容，转成带 \n 的单行字符串后写进 fleet_telemetry_config.ca，再用 check_server_cert.sh 验证。

文件 / 配置到底是什么用在哪里
tls/fullchain.pem服务端证书 + CA 链Fleet Telemetry / Caddy / proxy 的 TLS 服务端证书
tls/privkey.pem服务端 TLS 私钥和 fullchain.pem 配套使用，只留在服务器上
tls/chain.pem签发服务端证书的 CA 链通常作为下发配置里的 ca
fleet_telemetry_config.ca车辆用来验证 Fleet Telemetry 服务端证书的 CA 链下发给车辆；Certbot 场景通常填 chain.pem 内容
fleet-private-key.pem自己生成的虚拟钥匙私钥tesla-http-proxy 用它签名 Fleet Telemetry 配置请求和车辆命令
com.tesla.3p.public-key.pem从虚拟钥匙私钥导出的公钥托管到应用域名并注册给 Tesla；用户通过 _ak 添加虚拟钥匙后，车辆把这个公钥加入 keychain，用来验证后续签名
车辆客户端证书Tesla 车辆侧自带的客户端证书车辆连 Fleet Telemetry 时做 mTLS 客户端认证

不要混用：

  • ca 不填 privkey.pem，私钥永远不下发。
  • ca 不填虚拟钥匙公钥或私钥。虚拟钥匙只用于“私钥签名、公钥验签”，不参与 TLS 证书校验。
  • fullchain.pem 主要给服务端 TLS 使用；下发给车辆的 ca 通常取 chain.pem。
  • 如果 chain.pem 验证不通过或只提示 partial chain warning，就补齐 CA 链后再验证，直到 check_server_cert.sh 明确返回 The server certificate is valid.。
  • Cloudflare 在本文示例里只是 DNS-01 验证提供方，不是证书签发方；不要用 Cloudflare Origin CA 代替公开可信的服务端 TLS 证书链。

另外，/.well-known/appspecific/com.tesla.3p.public-key.pem 只能托管虚拟钥匙公钥，也就是 openssl ec -pubout 从 fleet-private-key.pem 导出的 com.tesla.3p.public-key.pem。这个文件应该是很小的 BEGIN PUBLIC KEY PEM；如果该 URL 返回 TLS 证书、CA 链、私钥、404 HTML 或重定向页面，Tesla 会下载到错误内容，可能报 Public key download failed ... too large。

推荐操作顺序

  1. 准备域名和公网入口，确定 Fleet Telemetry 真正暴露给车辆的 hostname 和 port。
  2. 创建 Tesla Developer 应用。
  3. 生成 prime256v1 虚拟钥匙对：私钥用于 tesla-http-proxy 签名，公钥用于托管和注册。
  4. 在 https://<应用域名>/.well-known/appspecific/com.tesla.3p.public-key.pem 托管公钥。
  5. 获取 Partner Authentication Token，并调用 Partner Account register 注册应用域名和公钥。
  6. 为 Fleet Telemetry 对外服务域名申请公开可信 TLS 证书。
  7. 部署 Fleet Telemetry 服务，使车辆可通过公网访问 hostname:port，并让 mTLS 终止在 Fleet Telemetry 服务本身，或终止在等价支持客户端证书校验的入口。
  8. 使用 check_server_cert.sh 校验将要下发给车辆的 hostname、port、ca。
  9. 引导用户通过 Tesla App 添加虚拟钥匙，让车辆把应用公钥加入 keychain。
  10. 部署 tesla-http-proxy，通过它调用 Fleet API 下发 fleet_telemetry_config；proxy 会用虚拟钥匙私钥签名配置。
  11. 查询配置状态，等待 synced=true，必要时查看 fleet_telemetry_errors。

三、组件说明

部署 Fleet Telemetry 需要依赖多个组件，包括：

  • Certbot: 用于申请 TLS 证书
  • Caddy: 作为反向代理，提供公钥托管，用于特斯拉访问；不要把 Fleet Telemetry 当作普通 HTTP 服务随意反向代理
  • tesla-http-proxy: 用于签名并转发 Fleet API 请求，向车辆下发 Fleet Telemetry 配置；也可以用 https://github.com/helloworlde/fleet-telemetry-proxy 在 Cloudflare Worker 部署替代
  • Fleet Telemetry: 车辆数据上报服务
  • MQTT: 用于接收并转发车辆上报的数据

四、创建域名解析

需要一个域名用于应用公钥托管和 Fleet Telemetry 服务器访问，如 fleet.example.com，并且解析到服务器的公网 IP 地址。在创建开发者应用和注册应用时，需要保证这个域名有效，并且域名中不能包含 tesla 关键字。

后续可以复用同一个域名部署不同服务，但要明确每个入口的调用方：

服务地址端口备注
Caddy 公钥托管fleet.example.com443需要公网可访问，用于 Tesla 后端和 Tesla App 获取公钥
Fleet Telemetryfleet.example.com12345需要公网可访问，车辆会直接连接该入口上报数据
tesla-http-proxy127.0.0.1 或内网地址4443不建议公网暴露，只供自己下发配置时调用

五、申请服务端 TLS 证书

以 Cloudflare 托管域名为例，使用 Certbot 申请服务端 TLS 证书。其他工具也可以申请，但需要能够拿到服务端证书链和中间证书链，供 Fleet Telemetry 服务和下发配置使用。这里的 TLS 证书用于 HTTPS/mTLS 服务端身份，不是虚拟钥匙。

本文的 Certbot 示例使用 dns-cloudflare 插件完成 DNS-01 验证。证书签发方是 ACME CA，Certbot 默认是 Let’s Encrypt；Cloudflare 在这里只负责托管 DNS 和写入验证 TXT 记录，不是这张证书的签发机构。如果显式改用其他 ACME CA，需要在文档和运维记录中写清楚实际签发方。

Certbot 生成的几个文件用途如下。后续文章统一使用 certs/tls/ 作为运行时挂载目录，申请完成后需要把 data/live/fleet.example.com/ 下的证书复制或软链接到该目录。

  • fullchain.pem: 服务端证书 + CA 链，配置给 Caddy、Fleet Telemetry 的 server_cert。
  • privkey.pem: 服务端 TLS 私钥，通常配置给 Caddy、Fleet Telemetry 的 server_key，必须保密。
  • chain.pem: CA 链，通常转成字符串后填到 fleet_telemetry_config.ca。
  • cert.pem: 仅服务端证书，一般不用在 Fleet Telemetry 配置里。

详细可以参考 DNS Plugins。

  • 创建 Cloudflare API Token

在 Cloudflare 创建一个 API Token，权限选择 “Zone - DNS - Edit”，然后选择对应的域名，然后将 token 写入到 data/cloudflare.ini 文件中

dns_cloudflare_api_token = xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
  • 申请证书

创建 docker-compose.yml 文件，内容如下：

services:
  certbot:
    image: certbot/dns-cloudflare:latest
    volumes:
      - ./data:/etc/letsencrypt/:rw
    command: >
      certonly
      --non-interactive
      --agree-tos
      --dns-cloudflare
      --dns-cloudflare-credentials /etc/letsencrypt/cloudflare.ini
      -d fleet.example.com

目录结构如下：

├── data
│   ├── cloudflare.ini
└── docker-compose.yaml

后续 Caddy、Fleet Telemetry 和 tesla-http-proxy 示例都会从 certs/tls/ 挂载证书。可以复制证书文件：

mkdir -p certs/tls
cp -L data/live/fleet.example.com/cert.pem certs/tls/cert.pem
cp -L data/live/fleet.example.com/chain.pem certs/tls/chain.pem
cp -L data/live/fleet.example.com/fullchain.pem certs/tls/fullchain.pem
cp -L data/live/fleet.example.com/privkey.pem certs/tls/privkey.pem

也可以创建软链接，但要确保 Docker 容器能读到软链接最终指向的真实文件。证书续期后，需要同步更新 certs/tls/ 中的文件并重启使用证书的服务。
